## Context

Two public-facing docs need corrections before 1.0.0:

1. `README.md` — contains a package name typo (`@open-product-oprim/cli` instead of `@open-product-primer/cli`) and uses `open-product-primer` as the primary command alias when `oprim` should be canonical.
2. `packages/cli/CHANGELOG.md` — auto-generated by release-please; contains commit-level entries from 0.7.2 → 0.14.0 but no narrative framing to give 1.0.0 context.

## Goals / Non-Goals

**Goals:**
- README is accurate and self-sufficient for a developer with no prior context
- CHANGELOG carries a human-written arc so 1.0.0 release notes land with context
- No code changes — this is purely a documentation pass

**Non-Goals:**
- Rewriting the README structure or repositioning oprim's value prop (that was BET-018)
- Automating changelog narrative generation
- Adding docs beyond README and CHANGELOG (e.g., a docs site)

## Decisions

**`oprim` as the primary alias throughout README**
The bin has two aliases: `oprim` and `open-product-primer`. All examples in the README currently lead with the long form. Flip every code block to use `oprim` and mention the long alias once in the npm package section only.

**Prepend narrative block above auto-generated CHANGELOG entries**
release-please owns the auto-generated section and will overwrite anything interleaved within it. Prepend a `## Development arc (0.x)` section above the first auto-generated `## [0.x.x]` heading so it survives future release-please runs.

**No version bump in this change**
The 1.0.0 tag is a separate action triggered after this change ships and passes review — not part of this task.

## Risks / Trade-offs

- release-please prepend position — if release-please ever regenerates the full file it may overwrite the narrative block. Mitigation: the block sits above the first auto-generated heading; release-please appends new entries at the top of the auto-generated section, so the narrative block will shift down safely.
