## Context

Oprim skill sessions (oprim-bet, oprim-pdr, oprim-review, etc.) and OpenSpec skill sessions (openspec-propose, openspec-apply-change, etc.) execute in isolation — there is no mechanism to inject relevant product decisions from `oprim/decisions/` into the active context. PDRs are durable artifacts that encode accepted decisions; surfacing them at the right moment prevents re-litigating settled choices.

The entire oprim and openspec skill content lives as string constants in `packages/cli/src/lib/install-agent.ts`. Installed skill files under `.claude/skills/` are generated by `oprim init/update` and are not the source of truth.

## Goals / Non-Goals

**Goals:**
- Surface relevant PDRs at the start of every oprim and openspec skill session **when the user has opted in**
- Gate the feature at `oprim init` and `oprim update` — disabled by default, no behavior change for users who don't opt in
- Use keyword matching against the active conversation to select which PDRs to surface
- Emit nothing when `oprim/decisions/` is empty or when no PDRs match — avoid noise
- Implement as a single new `oprim:context` skill invoked by all other skills (one place to change)

**Non-Goals:**
- Semantic / vector search over PDR content (keyword match is sufficient at current PDR volume)
- Auto-linking surfaced PDRs to bet or OpenSpec artifacts (a separate interaction)
- Surfacing PDRs on every agent prompt (only on explicit skill invocations)
- Modifying any existing PDR authoring or storage behavior

## Decisions

**Opt-in at init/update time, disabled by default**
The feature is gated by an explicit yes/no prompt during `oprim init` and `oprim update`. When opted in, `install-agent.ts` installs `oprim:context` and adds Step 0 to each skill; when opted out, skills are installed exactly as they are today. No runtime config file is needed — the presence or absence of the installed skill files is the state.

Defaulting to off avoids changing behavior for existing users on `oprim update` and lets early adopters trial the feature before it becomes standard. The kill criteria in BET-007 (>50% irrelevant PDRs surfaced) gives a clear threshold to flip the default or remove the feature.

*Alternative rejected*: Default on for all users — risks noise complaints before the matching quality is validated.

*Alternative rejected*: Persist opt-in preference in a config file — adds complexity; the installed file set is sufficient as implicit state. `oprim update` re-prompts so the user can change their mind.

**Single shared skill over per-skill embedding**
All skills call `oprim:context` as Step 0 rather than each skill duplicating the surfacing logic. This keeps the matching and display logic in one place and makes it easy to evolve (e.g., richer matching) without touching every skill.

*Alternative rejected*: Embed a "scan decisions/" step directly in each skill — high duplication, fragile to evolve.

**Keyword matching over semantic search**
The agent extracts topic keywords from the active conversation (current bet ID, capability name, filenames mentioned, high-level topics) and compares them against PDR filenames, title lines, and body text. This is deterministic, needs no external dependency, and is appropriate for the expected PDR corpus size (tens of records, not thousands).

*Alternative rejected*: Embedding-based vector search — adds an external dependency for marginal precision gain at this scale.

**Silent when empty or no match**
If `oprim/decisions/` is empty or no PDR matches the conversation keywords, `oprim:context` exits without output. Surfacing an empty state message would add noise and train users to ignore the output.

**Scope: oprim AND openspec skills**
The bet originally scoped this to oprim skills, but the user confirmed openspec skills should also invoke `oprim:context`. Decisions often affect what goes into a proposal, design, or spec — surfacing them at openspec time is equally valuable.

## Risks / Trade-offs

- [False positives — irrelevant PDRs surfaced] → Compact list format (title + status only) keeps noise low; user can ignore non-relevant entries; kill criteria in BET-007 catches sustained irrelevance
- [Latency] → Reading all PDR files adds a scan step; mitigated by bounded corpus size and local filesystem reads (no network)
- [Skill invocation chain confusion] → Users may be surprised that one skill calls another; mitigated by a brief "Checking product decisions…" prefix line before output

## Migration Plan

1. Add the `oprim:context` skill string constant to `install-agent.ts` (always compiled, conditionally installed)
2. Add the opt-in prompt to the `init` command flow in `commands/init.ts`
3. Add the opt-in prompt to the `update` command flow in `commands/update.ts` (re-prompts each time so users can change preference)
4. In the install logic: when `pdrSurfacing: true`, install `oprim:context` skill AND use skill string variants that include the Step 0 invocation; when `false`, install skill strings exactly as today
5. Run `oprim update` (opting in) in the project to verify the full install path
6. No data migration — `oprim/decisions/` exists and is ready to receive PDRs
7. Rollback: revert `install-agent.ts` and command changes, re-run `oprim update` (declining the opt-in)
