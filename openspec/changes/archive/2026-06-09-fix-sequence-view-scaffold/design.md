## Context

Three structural gaps were identified via git history comparison between what the CLI scaffolds for new installs and what exists in this repo's live workspace:

1. `oprim/scripts/generate-sequence-view.js` was never added to `init.ts` or `update.ts` — new projects never get it
2. `oprim/scripts/` directory is never created (`ensureDir` missing in `init.ts`)
3. `sequenceContent()` in `install-agent.ts` omits a final step to regenerate `oprim/sequence-view.md`, leaving the visual board stale after any sequence change; `oprimWorkflowsInline()` (Codex/Gemini) has no sequence workflow at all

## Goals / Non-Goals

**Goals:**
- Scaffold `oprim/scripts/generate-sequence-view.js` on `oprim init`
- Refresh the script on `oprim update` (same pattern as agent skill refresh)
- Remove the `js-yaml` dependency from the script; replace with an inline parser scoped to the `sequence.yaml` structure
- Wire a step 7 into `sequenceContent()` to run the script after board validation
- Add a `### Sequencing board` section to `oprimWorkflowsInline()` for Codex/Gemini parity

**Non-Goals:**
- Changing `sequence.yaml` format
- Auto-running the script on every bet write (that belongs to a future hook-based approach)
- Changing the mermaid output format

## Decisions

### Inline YAML parser, not js-yaml
The existing script used `require(path.join(__dirname, '../../packages/cli/node_modules/js-yaml'))` — a monorepo-only path that breaks outside this repo. Dropping js-yaml entirely and writing a ~40-line inline parser scoped to the four `sequence.yaml` buckets (`now`, `next`, `later`, `backlog`) removes the external dependency. The parser only needs to extract `id`, `title`, and `blocked_by` per item.

**Alternative considered**: Resolve js-yaml from the oprim CLI install path using `require.resolve` — fragile; breaks if global install path changes or user uses npx.

**Alternative considered**: Scaffold a `oprim/scripts/package.json` with js-yaml as a dep — adds friction (user must npm install in that dir).

### `update.ts` refreshes workspace scripts
`oprim update` currently only reinstalls agent skills. Scripts are part of the workspace the CLI owns, so `update` should also keep them current — the same way `init` writes templates. A single import of `ensureDir`/`writeFile` + `sequenceViewScriptTemplate` added before the agent skills loop accomplishes this silently.

**Alternative considered**: Scripts only written on `init`, not refreshed on `update` — means a user who first installed before 0.3.0 never gets the script unless they re-init.

### Skill runs the script, not the AI
The `/oprim:sequence` step 7 tells the agent to run `node oprim/scripts/generate-sequence-view.js` rather than having the AI regenerate the mermaid itself. The script is the single source of truth for diagram format; keeping format logic out of the skill prompt avoids drift if the format ever changes.

**Alternative considered**: AI generates mermaid inline — works everywhere but the format definition is split between the JS and the skill prompt text.

### Paths use `process.cwd()` not `__dirname`
The scaffolded script replaces the monorepo `__dirname`-relative paths with `process.cwd()`-relative paths (`path.join(process.cwd(), 'oprim/sequence.yaml')`). The script is always run from the project root.

## Risks / Trade-offs

- **Parser edge cases** — the inline YAML parser is tuned to the known `sequence.yaml` structure; unusual formatting (e.g. quoted titles with colons, deeply nested blocked_by) could misbehave. Acceptable given the format is machine-generated by the CLI itself.
- **update.ts now touches workspace files** — the first time a user runs `oprim update` after this lands, they'll get the script written. This is intentional and non-destructive (`writeFile` overwrites, same as templates).

## Open Questions

None — all decisions resolved.
