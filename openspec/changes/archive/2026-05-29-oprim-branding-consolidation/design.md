## Goals / Non-Goals

**Goals:**
- Every string a user reads in CLI output or `--help` says `oprim`, not "Open Product Primer"
- README template generated by `oprim init` leads with `oprim` so new project wikis are consistent
- Changes are pure string substitutions — no logic changes, no renames of source files or directories

**Non-Goals:**
- Renaming the npm package (`open-product-primer`) — that's a publishing/ecosystem concern for a separate bet
- Renaming the `primer/` workspace directory on disk — the directory path is the artifact address, not a brand
- Renaming the GitHub repo (`open-product-primer`)
- Updating the OpenSpec specs or design docs that refer to "Open Product Primer" (those are internal engineering docs, not user-facing)

## Decision

### Single substitution rule: "Open Product Primer" → `oprim` in user-visible strings

**Decision:** Replace the phrase "Open Product Primer" with `oprim` everywhere a user would read it — CLI descriptions, console output banners, and the README template. Do not change any file paths, variable names, or identifiers.

**Rationale:** The bet chose the simplest enforceable rule. One rule, no exceptions, no judgment calls at implementation time. Variable names like `primerDir` are not user-visible and renaming them would add noise with no user benefit.

**Alternative considered:** Replace with "Open Product Primer (oprim)" — adds parenthetical for discoverability. Rejected: the extra words re-introduce the two-name split the bet is trying to eliminate. Users already know the tool as `oprim`; the long form is not needed.
